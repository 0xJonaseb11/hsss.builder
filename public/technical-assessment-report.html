Summary for owners

The live app already delivers on-site pricing and order intake for four screen types. The gaps are in persistence, dashboard completeness, and registration deployment. I recommend extending the current codebase in phased releases: stabilize ordering and ship registration with a verified database layer, persist every order server-side, then connect AroFlo. A rewrite is not required.

HSSS has a deployed URL, a working pricing engine, screen configurators, branded order emails, and registration code that is written but not yet released with a versioned database schema. AroFlo API integration exists at prototype level. I have reviewed the repository and live dashboard and documented findings in the sections below.

1. Purpose of this document

This report documents the technical state of the HSSS builder ordering application after reviewing the live product, the developer handoff brief, and the full source repository. There is no formal SRS; the handoff brief and the code are the specification.

The platform functions today as an order intake and quoting tool. The target is a builder-facing CRM layer with persisted orders, account history, and a path into AroFlo. That transition can be done on the existing codebase.

2. What the platform is meant to do

Hydro Seal Shower Systems supplies semi-frameless shower screens to builders and renovators. The app allows builders on site to configure shower screens, see live prices, and submit orders to HSSS from a phone. It runs as a progressive web app installable from the browser. Two commercial modes apply: supply and install across south-east Queensland, and supply only nationwide.

From a CRM perspective, each builder is a client account. Each submission is a sales opportunity that should become a tracked order with a full audit trail. The handoff brief defines the immediate goal: test thoroughly, fix defects, and deploy the registration flow plus Supabase data layer without breaking live ordering. My plan extends that into a proper client and order management foundation.

3.2 What is sound to build on

Not everything needs replacing. These choices are worth keeping and extending:

  • Stack: React 18, Vite, PWA, Supabase, Vercel. Fits the product and team size.
  • Auth boundary: AuthWrapper renders App only after session and profile are ready. That separation should stay; auth UI should not move back into the order flow file.
  • Data hooks: useBuilderProfile and useSiteContacts in useBuilderDB.js follow a normal React pattern (fetch, mutate, error state). New entities (orders, quotes) should use the same hook style.
  • Serverless API folder: api/aroflo.js already keeps third-party secrets off the client. Order submit and AroFlo sync should follow this pattern.
  • Domain logic: calcPrice(), screen configurators, and measurement validation encode real product rules. The logic is correct to preserve; it needs extraction into importable modules, not a rewrite.
  • Inline UI components: Small presentational pieces (Btn, MeasurementInput, ColourPicker) are consistent within the app. They should move to a shared folder, not be redesigned.

3.3 What should not continue as-is

These patterns are typical of a rapid prototype. They are not a foundation to keep adding features on top of indefinitely:

  • Monolithic App.jsx: Pricing, diagrams, navigation, and every screen in one file makes changes risky and review difficult. New work should not add more inline components here.
  • String-based navigation: screen state with a large switch block works for a demo but does not scale. Deep links, password reset, and order detail views will need a router or equivalent route table.
  • Split auth implementation: Live auth lives in AuthWrapper.jsx, but App.jsx still contains ~400 lines of unused LoginScreen, RegisterScreen, and SplashScreen. Profile saves are duplicated: hooks exist in useBuilderDB.js while AuthWrapper calls Supabase directly.
  • Duplicated constants and utilities: REGIONS is defined in both App.jsx and AuthWrapper.jsx. Brand colours appear in three files. fmtDate exists in emailService.js but not in QuoteSummary, which shows raw ISO dates (BUG-22).
  • Duplicated diagram code: Screen previews are SVG React components in App.jsx. Email diagrams are separate SVG string builders in emailService.js. Same geometry, two implementations; they can drift.
  • Client as system of record: Order submit and email send run in the browser with no API in between. That is an architectural choice to replace, not patch.
  • No schema in repo: Without supabase/migrations/, RLS and table design cannot be reviewed or reproduced from git.
  • No automated checks: No ESLint, no tests, no TypeScript on pricing or order shapes. Pricing regressions are likely without calcPrice unit tests.
  • Inline styles only: Acceptable for the current UI volume. A staff admin portal on the same codebase will need shared layout and form components.

3.4 Recommended organization

Evolve toward this layout as features ship. Order of extraction: pricing and shared utils first, then order flow screens, then auth cleanup. 

src/
  lib/                  pricing.js, constants.js, dates.js, brand.js
  components/           Btn, Input, Card, MeasurementInput, ...
  hooks/                useBuilderDB.js (and future data hooks)
  features/
    auth/               AuthWrapper, ResetPassword
    dashboard/
    orders/             quote setup, summary, submit, history
    screens/            FrontReturn, FixedPanel, QuickQuote, ...
  email/                templates, diagram renderers (shared where possible)
api/
  submit-order.js       persist + notify
  aroflo.js             existing proxy
supabase/
  migrations/           versioned schema + RLS

Coding conventions going forward: one source for product constants; Supabase access only through hooks or server routes, not ad hoc calls in UI files; pricing and validation importable from both client and API; delete dead code when auth migration is complete; add tests when touching calcPrice or panel rules.

Verdict: The codebase reflects a working prototype, not a finished application structure. Stack and domain logic are worth keeping. File organization, data access consistency, and server-side order handling need deliberate refactoring as part of delivery, not a separate rewrite project.

4. Live product assessment

4.1 What works in production today

Four screen types are fully configurable: front and return, front only, splayed, and fixed panel. Custom panel and standalone custom order flows exist. Live pricing reflects supply and install versus supply only tables, colour surcharges, door size upgrades, sliding premiums, and radius corner fees on fixed panels. Builders can schedule delivery dates, label multi-screen jobs, save site contacts, run quick quotes, and edit or remove screens before submit. Order emails include branded HTML and per-screen diagrams. Measurement inputs accept free numeric entry with min and max enforced on blur, matching the handoff specification.

4.2 Current dashboard (live screenshot)

The home screen should show who is logged in, recent orders, and clear next steps. Today it routes to configurators only. Gaps are listed as BUG-09 through BUG-11 in section 5.

4.3 Built in code but not safely in production

A three-step registration flow exists in AuthWrapper.jsx and captures company name, ABN, contact details, service type, region, and address. Site contact persistence is wired to a site_contacts table through useBuilderDB.js. However, no SQL migrations appear in the repository, row level security policies cannot be reviewed from source control, and registration has not been validated on a staging deploy against production data. Password reset emails reference a /reset-password route that does not exist. PWA icon files are now present in the repository under public/; confirm they are deployed correctly on production. These items block a confident registration release.

5. Bugs and defects identified in review

These are specific issues traced through the source files and the live dashboard at hsss-builder-app.vercel.app. Each entry describes what a builder or staff member actually experiences, what the code does, and where it lives. Severity is based on whether a job, a price, or account data can be wrong or lost.

Order submission and email

BUG-01 Critical App.jsx submitOrder (lines 1679 to 1683)

What happens: Builder taps Submit Order. The app calls sendOrderEmail via EmailJS from the browser. Nothing is written to Supabase. There is no orders table in the repository.

Impact: If EmailJS is down, over quota (~200 emails/month on free tier), or misconfigured (VITE_EMAILJS_* env vars in emailService.js lines 7 to 9), the job may never reach bradley@hsss.net.au and HSSS has no backup record.

BUG-02 Critical App.jsx lines 1679 to 1681

What happens: setScreen("quoteSubmitted") runs immediately. sendOrderEmail runs after. The builder sees "Order Submitted!" and a green success layout while email status is still "sending".

Impact: On failure, the headline still says submitted. The error only appears in a smaller card below. Easy to miss on a phone.

BUG-03 High App.jsx QuoteSubmitted line 1625

What happens: When email fails, the UI tells the builder: "Your order details are saved."

Impact: That is incorrect. Nothing is saved anywhere. The builder may assume HSSS has the order and not call sales.

BUG-04 High App.jsx CustomOrderScreen line 1218

What happens: Custom measure requests show the same false message: "Order saved but email failed to send" when EmailJS fails.

Impact: Same as BUG-03 for non-standard jobs that often need the most staff attention.

BUG-05 Medium emailService.js lines 481 to 485

What happens: Before send, payload size is checked. Above ~48KB, all <img> diagram tags are regex-stripped from the HTML email.

Impact: Large multi-screen orders still submit. HSSS receives an email with specs but no diagrams. The builder is not told diagrams were removed.

BUG-06 Medium emailService.js sendCustomOrderEmail

What happens: Standard orders include company name, contact, email, and phone from builderProfile. Custom order emails only send address, description, measure date, and optional site contact typed on that screen.

Impact: HSSS may not know which registered builder sent a custom request unless they recognize the address.

BUG-07 Low App.jsx handleQuoteSubmit, LocationLabels

What happens: Orders with two or more screens go to the location label screen. Single-screen orders call submitOrder([]) directly with no location prompt.

Impact: A one-screen job to "Main bathroom" never gets a room label in the email unless the builder puts it in free-text notes.

BUG-08 Low App.jsx submit buttons on summary and location screens

What happens: Submit Order has no loading or disabled state while email is sending. A fast double tap can fire two EmailJS sends for one job.

Impact: Duplicate emails to HSSS; possible duplicate processing if staff treat each as a separate job.

Dashboard and navigation

BUG-09 High App.jsx lines 1695 to 1700

What happens: Dashboard defines six actions. onNavigate handles newOrder, quickQuote, customOrder, and contact only. Actions myQuotes and orders fall through to setScreen("dashboard"): same screen, no feedback.

Impact: Visible on live app (see Figure 1). Two tiles have no handler (BUG-09, BUG-10).

BUG-10 High useBuilderDB.js lines 228 to 259

What happens: useBuilderOrders sets loading, then always sets orders to []. No Supabase query.

Impact: Even after wiring navigation for "Order history", there is no data layer to connect.

BUG-11 High App.jsx line 1640; AuthWrapper.jsx line 716

What happens: signOut is passed into HsssBuilderApp but never referenced in the component. Dashboard shows "Welcome back" with no company name and no logout control.

Impact: Shared phones on site cannot switch builder accounts without clearing browser data. builderProfile is available but not displayed.

Auth and registration

BUG-12 High AuthWrapper.jsx line 276

What happens: Forgot password sends reset email with redirectTo: ${origin}/reset-password. No component, route, or page handles that path in the app.

Impact: User clicks email link and lands on a blank or Vercel 404. Password reset flow is broken end to end.

BUG-13 Medium AuthWrapper.jsx lines 214 to 221, 89 to 101

What happens: On signup, if profile insert fails (common when email confirmation is required), profile JSON is stored in localStorage under hsss_pending_profile until next login.

Impact: Different browser or cleared storage loses registration details. Builder completes signup but profile may never reach Supabase silently.

BUG-14 Medium AuthWrapper.jsx needsProfile UI, lines 454 to 511

What happens: Profile completion shows a 3-step progress indicator but only two form steps (company details, then address). Step 3 from signup does not exist on this path.

Impact: Misleading progress UI for builders finishing a partial account.

BUG-15 Medium Repository: no supabase/migrations

What happens: Code expects builders and site_contacts tables. No SQL migrations or RLS policy files are in the repo.

Impact: Registration deploy cannot be reviewed for "builder A cannot read builder B data" before production. Schema drift between environments is likely.

Pricing and screen configurators

BUG-16 Medium App.jsx calcPrice lines 99 to 103

What happens: Fixed panel uses smallMax: 985 and largeMin: 1035. Width 986 to 1034 hits the else branch and gets the small price ($704.58 supply and install / $504.21 supply only).

Impact: Example: 1000mm panel priced as small tier. Handoff brief flags this range. Brad needs to confirm intended price or builders are quoted wrong.

BUG-17 Medium App.jsx QuickQuote lines 1327 to 1331

What happens: Front and Return builder blocks sliding doors under 1200mm front width (slidingTooSmall). Quick Quote allows sliding toggle on front return with no minimum check and still calls calcPrice with isSliding: true (+$150).

Impact: Builder gets a price for a configuration that cannot be added to a real order. Quick Quote and New Order disagree on the same inputs.

BUG-18 Medium App.jsx FixedPanelBuilder lines 1002 to 1017

What happens: Walkthrough opening under 600mm shows an amber warning ("may not be fit for purpose"). Add to Order stays enabled.

Impact: Unbuildable or unsafe openings can still be submitted to HSSS.

BUG-19 Medium App.jsx line 1645

What happens: customerType is set once from builderProfile.service_type when the app loads. No toggle in ordering or quick quote flows.

Impact: Supply-only builder sees supply-and-install prices (or reverse) if profile service type is wrong or outdated. Affects every calcPrice call in the session.

BUG-20 Low App.jsx COLOURS line 22

What happens: Handoff product list says "Brushed Gold". App label is "Brushed Brass" with the same surcharge tiers.

Impact: Orders and emails may not match HSSS catalogue naming. Product decision, not a crash.

6. Proposed fixes

Mapped to section 5 bug IDs. Implementation order: submission and persistence, dashboard and auth, pricing corrections, then UI and security hardening. The existing sendOrderEmail import in App.jsx stays stable during transition; the server route can call the same email logic internally.

Order submission and email

Addresses BUG-01, BUG-02, BUG-08

Server-side submit endpoint. Add POST /api/submit-order (Vercel serverless or Supabase Edge Function). Payload: builder id, quote fields, screens JSON, location labels. Handler order: validate auth session, validate pricing server-side against the same calcPrice rules, insert row into orders + order_items, then call email send. Return { orderId, emailSent }.

Client change. submitOrder awaits the API response before navigating. Show a loading state on Submit Order (disabled button + "Sending..."). Success screen displays order reference number. If email fails but DB insert succeeded, show: "Order #1042 received. Email notification delayed. HSSS has your order."

Idempotency. Client generates a submission_id UUID per attempt. Server rejects duplicate submission_id to block double-tap sends (BUG-08).

Addresses BUG-03, BUG-04

Remove all copy that says "saved" unless a database row exists. Replace with accurate states: sending, received (order id), email failed (with sales phone links). Apply to QuoteSubmitted and CustomOrderScreen success views.

Addresses BUG-05

Move email build to the server where payload limits are easier to manage. If diagrams must be stripped, include a line in the builder confirmation: "Diagrams omitted from email due to size; full spec stored on order #1042." Long term: store diagram PNGs in Supabase Storage and link from the email instead of inline base64.

Addresses BUG-06

Pass user and builderProfile into CustomOrderScreen. Extend sendCustomOrderEmail to use the same builder header block as standard orders (company, contact, email, phone).

Addresses BUG-07

7. Technical direction

Target product behaviour

Builder logs in, sees company name and recent orders, configures screens with live pricing aligned to the rate card, submits once, and receives an order reference. HSSS gets email with diagrams. The same record syncs to AroFlo as a task with resolved assemblies. Order history supports repeat jobs without re-entry.

Later phase: HSSS staff view for registrations, submissions, and custom panel flags. Pricing maintained via configuration or database table rather than scattered constants in App.jsx.

Target architecture

Client: React PWA split into feature modules (orders, screens, pricing). Auth remains in AuthWrapper. Pricing logic extracted to a shared module used by client and server.

API (Vercel serverless): Order submit, notification send, AroFlo sync. Secrets server-side only.

Data (Supabase): builders, site_contacts, orders, order_items, aroflo_client_map when integration is ready. Migrations versioned in the repository. RLS verified before registration goes live.

Integrations: Email for notifications; AroFlo for job management; Supabase Storage for diagram assets if inline email payloads remain too large.

Production-ready criteria

Every submission creates a database record. Email is a notification channel, not the system of record. Builders have order history and sign-out. Password reset works. Registration ships with tested RLS. Pricing edge cases documented and covered by tests. Dashboard shows account context. AroFlo wired for standard stock orders first. Codebase modular enough to maintain as a single long-running product.

8. Data and integration gaps

Orders are not persisted today. Builder profiles and site contacts are the only CRM entities in Supabase. EmailJS carries a monthly volume cap and payload limits on large orders. AroFlo proxy credentials need hardening before production CRM use. Full detail is in section 5; fixes are in section 6.

10. Recommendation: evolve, do not rebuild

A full rewrite would delay value and reintroduce pricing risk for builders already using the live URL. The existing engine encodes domain knowledge: surcharge rules, screen geometry, diagram generation, and email layout. The path forward is to add versioned schema, server-side order capture, and phased AroFlo integration, while restructuring the codebase per sections 3.3 and 3.4 (extract modules from App.jsx, consolidate data access, remove dead auth code).

Auth and database changes will go through Vercel preview deploys before production merge.

11. Delivery plan

Phase 1: Stabilize and ship registration

Typical focus over the first few weeks. Obtain Supabase, Vercel, and EmailJS access. Run full mobile QA on iOS Safari and Android Chrome across all screen types and both pricing modes. Fix priority defects. Export Supabase schema and row level security into versioned migrations in the repository. Deploy registration and site contacts on a preview branch, then production after sign-off. Add missing PWA assets, password reset page, and sign-out in the main app. Hide or properly stub non-functional dashboard tiles until data exists behind them.

Phase 2: Order persistence and CRM foundation

Typical focus over the following weeks once stabilization is complete. Introduce orders and order_items tables linked to builders. Move submission to a server-side function that writes to the database first, then sends email with retry logic. Implement order history and saved quotes in the dashboard. Replace client-only EmailJS as the system of record with a server-side mail provider when volume requires it. Redesign the dashboard to show builder identity, recent orders, and clear next actions.

Phase 3: Engineering hardening

Typical focus over subsequent weeks as features stabilize. Extract lib/pricing.js and shared constants per section 3.4. Move screen builders and order flow into features/. Wire AuthWrapper through useBuilderProfile instead of direct Supabase calls. Add unit tests on calcPrice() and panel formulas from the handoff document. Introduce linting and CI.

Phase 4: AroFlo and staff workflows

Typical focus once order persistence is proven in production. Secure the AroFlo proxy behind authenticated sessions. Create tasks on order submit using the existing HMAC integration pattern. Map builders to AroFlo clients once the full client list issue is resolved with AroFlo support. Apply assembly resolver logic for line items. Run parallel email parsing only as fallback until API submission is proven on real orders.

Phase 5: Scale and admin capability

Ongoing work as the platform matures. HSSS staff portal for registration approval, order review, and pricing overrides. Builder notifications on status changes. Slider screen type when the panel formula is confirmed. Operational analytics on order volume and configuration mix. Custom production domain and monitoring.

12. Scalability approach

Vercel and Supabase are sufficient for thousands of monthly orders without platform migration. Scaling effort belongs in application design: server-side validation prevents price tampering, indexed order tables support fast history queries, cached AroFlo product lookups respect the two thousand calls per day limit, and structured logging plus error tracking surface failures before builders report them.

Configuration-driven pricing and region rules allow commercial updates without redeploying the entire frontend. Modular structure keeps the codebase maintainable as a single long-running product. Migrations in source control make database state reproducible; that versioning is not in the repository today.

15. Expected outcomes

AroFlo integration reduces manual re-entry. These outcomes follow from completing the phases in section 11 on the existing codebase.

Closing statement for owners

The application has a working core: pricing, screen builders, and email-based order delivery. The work ahead is to make submissions reliable, complete the dashboard and registration layer, and connect to AroFlo. The live app remains the base; the plan is extension and hardening, not a rebuild.
