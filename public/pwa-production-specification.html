Specification

HSSS Builder PWA

Prepared by Ramil Huertas  |  June 2025
Stack: Next.js, TypeScript, Tailwind CSS, Supabase, Vercel

Builder-facing PWA: register, log in, reset password, sign out, price screens, save quotes, submit orders. Orders go to Supabase first, then email to HSSS. Installable from the home screen (manifest + service worker).

1. Features

FeatureDetail
Registration Email and password (Supabase Auth). Profile: company name, ABN, contact, service type, region, address. Saved to builders.
Login Email and password. Session cookies via @supabase/ssr. Routes behind auth redirect to login.
Forgot password Supabase sends reset link. New password at /auth/reset-password.
Sign out Header button on dashboard. Ends session, back to login.
Quotes Live pricing, screen config (Front & Return, Front Only, Splayed, Fixed Panel, Quick Quote). Drafts in Supabase: list, open, delete.
Orders Submit saves to DB, emails HSSS, shows order ref to builder. Order list + detail. Re-order copies into a new quote.
Dashboard Company name, New Quote, My Quotes, My Orders, sign out.

2. Email notifications

Order emails run on the server (Next.js API route), not in the browser. Row is written to the database before the email is sent.

EventRecipient
Order submittedHSSS inbox (e.g. bradley@hsss.net.au): job details, screens, prices
Order confirmationBuilder: order reference and summary
Password reset / verifyUser: Supabase Auth templates

Push and SMS are out of scope for the first release.

3. Email providers

Auth mail stays on Supabase. Order mail is one of the options below, wired through a shared server module so the provider can be swapped via env vars.

ProviderNotes
Google SMTP smtp.gmail.com, port 587, App Password. Works with Google Workspace on @hsss.net.au. Nodemailer on the server. No third-party signup if mail already lives on Google.
Microsoft 365 SMTP smtp.office365.com. Same Nodemailer setup as Google.
Resend REST API. Free tier caps monthly sends; paid tiers above that.
SendGrid REST API. Free tier for low volume.
Postmark REST API. Paid; no real free tier.
Amazon SES Cheaper at volume. Needs AWS account and domain verification.
EmailJS Used in the current app from the client. Drop it for the rebuild; orders need server-side send.

Google Workspace on @hsss.net.au: use SMTP. Otherwise Resend or SendGrid. Env vars for SMTP: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, ORDER_EMAIL_FROM, ORDER_EMAIL_TO. API providers use their own keys instead.

4. PWA install

  • Manifest: app name, icons 192 and 512 (maskable), display: standalone, theme #003A70, start URL points at dashboard.
  • Service worker (Serwist or next-pwa): cache static files; always fetch API and auth over the network.
  • Android: install prompt after first login. iOS: short how-to (Share, Add to Home Screen).
  • Icons in public/: favicon, 192, 512, Apple touch 180.
  • /offline page when there is no connection. Drafts sit in Supabase, not on the device.
  • HTTPS on Vercel. Check Lighthouse installability before go-live.
  • Prompt to reload when a new build is deployed.

5. Stack

LayerTechnology
FrontendNext.js (App Router), TypeScript, Tailwind CSS
Auth + databaseSupabase (Auth, Postgres, RLS, migrations in repo)
HostingVercel, including /api/orders/submit
Order emailGoogle SMTP, Resend, SendGrid, or Postmark (server-side)
PWASerwist or next-pwa

Submit path: client posts to API route, server checks session and pricing, inserts orders, sends email, returns reference. Pricing rules move over from the current App.jsx into lib/pricing.
